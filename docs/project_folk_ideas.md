# このプロジェクトの目的と背景
- このプロジェクトは、複数の学生集客イベントを企画運営する事業会社にて、モダンで集客力の高いLPを作成するものです。
- それにあたりイベントごとに日時や場所などの詳細が異なるものの、現場は非エンジニアであることからソースコードの直接編集を行うにはハードルが高く、またUI/UXの微調整を行いたい(CTAボタンの配置や配色、画像の差し替えや追加などコンテンツがイベントによって異なる場合がある)というニーズに応えるべく、当初は「Manus」のWeb Static機能を使用して制作を考えていた。
- 「Manus」は非エンジニアでもGUI操作でコンテンツの変更ができたり、プロンプトベースでの調整が図れる他、カスタムドメインの設定ができることから当社のニーズにマッチしていると考えられた。
- しかし、追加要件としてこのLPは同様のコンテンツのものをURLのクエリを変えて複数用意する必要がある、という要件が出てきた。
- これは事後的にどの広告媒体から流入したものであるかを分析するための仕様で、URLの一部に`/&from?=free`、`/&from?=saiyocom`などのクエリを付加したQRコードを広告媒体別に添付することで、集客の流入経路を特定する手法。サイトの中身は同じだがURLのクエリだけが異なるという仕組みである。
- これを「Manus」で実現するためには採用媒体の数だけプロジェクトを生成しなければならず、コストがかかってしまう。
- また、「Manus」でGUI操作による変更ができるとはいえ、自由度はコーディングによって拡張した方が高まる。
- さらに「Manus」はGithubのmainブランチの反映にすらクレジットを消費するため、非常にコストが高い。

# 狙い
- そこで、Github Pagesを使って独自ドメインによるLPを作成して公開するアプローチを検討している。
- 現在の`/contents-manager`機能を拡張することによって、非エンジニアでも簡単に、かつ自由度高くコンテンツの編集が可能になるのではないかと期待している。
- UIについても、5-10パターン程度のカラーパレットを用意することによってバリエーション豊かなUIが実現できるのではないかと考えている。

# 懸念点
- Github Pagesはそれ相応の制約条件もあることから、果たして期待しているほど簡単に運用できるものなのか怪しい。
- 特に画像差し替えや追加などは頻繁に起こるニーズとして想像できる。
- また、Manusを前提としたプロジェクトであったためGithubに簡単に移植あるいはブランチをフォークできるものなのか(mainブランチはManus、サブブランチはGithub Pagesなどの運用ができると保険が効いてなお良い)、甚だ疑問である。
